# Jacla Portal マルチテナントSaaS再設計案

更新日: 2026-05-04

## 1. 背景

Jacla Portal は現在、単一団体向けのポータルとして構築されている。
今後は学内の複数団体で共通基盤として利用しつつ、各団体が独自の運用ルール・役職・楽器・機能設定を持てる構成へ移行する。

想定している利用形態は以下の通り。

- 学内の複数団体が同一システムを利用する
- 同一ユーザーが複数団体に所属できる
- 団体ごとに役職、担当、楽器、利用機能、入力項目が異なる
- プラットフォーム全体は中央管理者が統制する
- 各団体には団体単位の管理者 `SV` を置き、自団体の設定を管理させる

本書では、この要件を満たすためのアーキテクチャ方針と移行方針を定義する。

---

## 2. 再設計の目的

### 2.1 到達目標

1. 1つのアカウントで複数団体に所属できること
2. 団体ごとに異なるシステム権限を持てること
3. 団体ごとに異なる役職・楽器・運用ルールを持てること
4. 共通コードベースのまま複数団体を運用できること
5. プラットフォーム管理者が全団体を統制できること

### 2.2 非目標

現時点では以下は対象外とする。

- 外販向けの課金・請求機能
- 団体ごとの別デプロイ
- 団体ごとの独自コード分岐
- 団体ごとの独自 SQL / 独自 RLS を持つ構成

---

## 3. 設計原則

### 3.1 人と所属を分離する

`ユーザー本人` と `団体内での立場` を分離する。

- `profiles`: 人そのもの
- `organization_memberships`: どの団体に所属しているか
- `organization_member_system_roles`: 団体内でのシステム権限
- `organization_member_positions`: 団体内での役職
- `organization_member_instruments`: 団体内での担当楽器

これにより、同じ人が団体 A では管理者、団体 B では一般部員、という状態を表現できる。

### 3.2 システム権限と部内役職を分離する

以下は別概念として扱う。

- システム権限: 画面や機能を操作できるか
- 部内役職: 部長、会計、PA 長などの肩書き

例:

- `Org SV` はシステム権限
- `部長` は部内役職

両者は独立して付与可能にする。

### 3.3 カスタマイズは設定駆動に寄せる

各団体ごとの差分は、原則として以下で吸収する。

- マスターテーブル
- 設定テーブル
- 機能フラグ
- 追加項目定義

コード分岐で団体差分を吸収し始めると、運用不能になるため避ける。

### 3.4 権限判定は関数に集約する

RLS や API 内の権限判定は、文字列比較を散在させず、共通関数に寄せる。

例:

- `is_administer(user_id)`
- `is_org_member(user_id, organization_id)`
- `has_org_permission(user_id, organization_id, permission_key)`

## 4. 目標アーキテクチャ

## 4.1 論理レイヤ

```text
Platform Layer
  - 全団体管理
  - 団体作成/停止
  - 権限テンプレート管理
  - 監査

Organization Layer
  - 団体別設定
  - 団体別メンバー管理
  - 団体別役職/楽器/機能管理
  - 団体別データ運用

User Layer
  - 個人プロフィール
  - 所属団体切替
  - 個人タスク/提出/通知
```

## 4.2 URL 構成

以下の 3 階層に整理する。

```text
/platform/*         プラットフォーム全体管理
/me/*               個人共通ページ
/o/[orgSlug]/*      団体別ポータル
```

例:

- `/platform/organizations`
- `/platform/audit-logs`
- `/me/profile`
- `/me/organizations`
- `/o/jacla/events`
- `/o/keion/bands`

この構成により、`今どの団体を見ているか` をルーティング自体が表現できる。

## 5. 権限モデル

## 5.1 プラットフォーム権限

| 権限キー             | 役割                                                 |
| -------------------- | ---------------------------------------------------- |
| `administer`      | 全団体の統制、設定、監査、停止、機能テンプレート管理 |
| `platform_support` | 閲覧補助、運用補助、限定的な保守対応                 |

`administer` は最上位権限であり、全組織に対して横断的に操作できる。

## 5.2 団体システム権限

| 権限キー                       | 役割                                               |
| ------------------------------ | -------------------------------------------------- |
| `org_sv`                     | 団体の最上位管理者。設定、メンバー、役職、機能管理 |
| `org_admin`                  | 団体の管理者。主要管理機能へのアクセス             |
| `module_admin_events`        | イベント管理                                       |
| `module_admin_bands`         | バンド・提出管理                                   |
| `module_admin_announcements` | お知らせ管理                                       |
| `module_admin_equipment`     | 機材管理                                           |
| `module_admin_forms`         | フォーム管理                                       |
| `module_admin_members`       | メンバー管理                                       |

## 5.3 団体内役職

団体内役職は団体が自由に定義できるようにする。

例:

- 部長
- 副部長
- 会計
- PA 長
- 照明長
- 広報
- Web 幹事

役職は表示・人事上の概念であり、必ずしもシステム権限と一致しない。

## 5.4 想定ユースケース

### ケース A

- ユーザー A
- Jacla では `org_sv`
- 軽音部では `member`

### ケース B

- ユーザー B
- Jacla では `member`
- 軽音部では `module_admin_events`
- どちらでも `Gt.` を担当

### ケース C

- ユーザー C
- Jacla では `PA 長` だが、システム権限は `module_admin_equipment` のみ

## 6. データモデル方針

## 6.1 コアテーブル

### 個人情報

- `profiles`
  - ユーザー本人の共通プロフィール
  - 表示名、氏名、メール、アイコンなど
- `profile_private`
  - 個人の機微情報
  - 学籍番号、電話番号、生年月日など

### プラットフォーム管理

- `platform_user_roles`
  - ユーザーに対する全体権限
- `organizations`
  - 団体マスタ
  - `id`, `slug`, `name`, `status`, `created_at`
- `organization_memberships`
  - 所属情報
  - `user_id`, `organization_id`, `membership_status`, `joined_at`

### 権限・役職・楽器

- `organization_system_roles`
  - 団体で利用できるシステム権限定義
- `organization_member_system_roles`
  - ユーザーへの団体権限付与
- `organization_positions`
  - 団体独自の役職定義
- `organization_member_positions`
  - ユーザーへの役職付与
- `organization_instruments`
  - 団体独自の楽器/パート定義
- `organization_member_instruments`
  - ユーザーへの楽器付与

### 機能設定

- `organization_modules`
  - 団体ごとに利用するモジュールの ON/OFF
- `organization_module_settings`
  - モジュールごとの詳細設定
- `organization_custom_fields`
  - 追加項目定義
- `organization_audit_logs`
  - 団体内変更ログ

## 6.2 既存ドメインテーブルへの方針

以下の既存テーブルには原則 `organization_id` を追加する。

- `events`
- `bands`
- `band_members`
- `songs`
- `announcements`
- `equipment_items`
- `equipment_instruments`
- `equipment_item_logs`
- `equipment_instrument_logs`
- `feedbacks`
- `forms`
- `external_link_pages`
- `site_settings`
- `update_logs` のうち団体別運用にしたいもの

## 6.3 廃止または再配置する概念

現在 `profiles` 側にある以下の情報は、全体属性ではなく団体属性へ移す。

- `leader`
- `crew`
- `part`
- `muted`

移行先の例:

- `leader` -> `organization_member_system_roles`
- `crew` -> `organization_member_positions` または団体独自分類テーブル
- `part` -> `organization_member_instruments`
- `muted` -> `organization_memberships` または団体別メンバー状態

## 7. カスタマイズモデル

## 7.1 団体が自由に変えられる範囲

- 役職定義
- 楽器/パート定義
- 利用モジュール
- モジュールごとの表示文言
- プロフィール必須項目
- バンド提出項目
- イベント入力項目
- リンク集
- 一部ワークフロー設定

## 7.2 団体が変えられない範囲

- 認証基盤
- RLS の基本構造
- プラットフォーム全体の監査方針
- デプロイ方式
- 団体独自コードの注入

## 7.3 実装指針

`楽器` `役職` `カテゴリ` などの差分を Postgres ENUM で持つのは避ける。
複数団体対応ではマスターテーブル方式の方が変更に強い。

---

## 8. RLS / API 方針

## 8.1 基本方針

- 全ての団体データは `organization_id` でスコープする
- API はクライアント側判定を信用しない
- 管理系 API は必ずサーバー側で権限再判定する
- Storage もパス規約で団体分離する

## 8.2 補助関数の例

- `is_administer(uid uuid)`
- `is_platform_support(uid uuid)`
- `is_org_member(uid uuid, org_id uuid)`
- `has_org_role(uid uuid, org_id uuid, role_key text)`
- `has_org_permission(uid uuid, org_id uuid, permission_key text)`

## 8.3 Storage パス方針

以下のようなプレフィックスで管理する。

```text
avatars/{userId}/...
org/{organizationId}/announcements/...
org/{organizationId}/events/...
org/{organizationId}/forms/...
```

## 10. 移行方針

## フェーズ 0: 先行修正

- 管理 API の認可漏れを修正する
- 権限判定をフロント依存にしない

## フェーズ 1: コア追加

- `organizations`
- `organization_memberships`
- `platform_user_roles`

を追加する。

まずは Jacla を最初の `organization` として登録する。

## フェーズ 2: ルーティング再編

- `/platform/*`
- `/me/*`
- `/o/[orgSlug]/*`

へ再編する。

この段階で `現在選択中の団体` を UI と URL で一貫して扱う。

## フェーズ 3: 権限再編

- 既存の `leader`, `position`, `part` 依存を整理する
- 団体スコープの権限・役職・楽器テーブルへ移行する

## フェーズ 4: ドメインテーブルの団体化

主要業務テーブルへ `organization_id` を追加し、既存データを Jacla 配下へ寄せる。

## フェーズ 5: RLS 再構築

`organization_id` ベースで RLS を組み直す。

## フェーズ 6: 団体追加

軽音部など次団体を追加し、機能・役職・楽器を設定で差し替える。

---

## 11. 先に確定しておくべき設計判断

以下は早めに固定した方がよい。

1. ルーティングは団体スラッグを URL に含める
2. `システム権限` と `部内役職` は分離する
3. 団体差分は ENUM ではなくマスターテーブルに寄せる
4. 既存の主要ドメインテーブルは `organization_id` 必須にする
5. カスタマイズは設定駆動に限定し、団体別コード分岐は避ける

---

## 12. 当面の実装優先順位

1. 認可漏れのある管理 API の是正
2. `organizations` / `organization_memberships` 追加
3. 団体切替 UI の導入
4. 権限モデルの再定義
5. `organization_id` の付与
6. RLS 再構築
7. 団体別設定画面の追加

---

## 13. まとめ

今回の再設計は、単なる「複数部活対応」ではなく、`中央統制つきマルチテナント基盤` への移行である。

設計の中心は以下の 3 点にある。

- 人と所属を分離する
- システム権限と役職を分離する
- 団体差分を設定駆動で吸収する

この前提を守れば、Jacla と軽音部を同じ基盤で運用しつつ、将来的な団体追加にも耐えられる。
